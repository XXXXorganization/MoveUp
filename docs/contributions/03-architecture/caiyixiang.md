##
遇到的问题和解决
1. 问题：
    不了解什么是数据库迁移（migration），以及它如何解决版本控制问题。
    不清楚如何将迁移脚本集成到 Node.js 项目中，如何编写、执行和管理这些脚本。
    担心手动修改数据库后，团队成员无法同步，环境不一致，且无法回滚错误的结构变更。
解决：

    1）理解数据库迁移脚本是一种版本化数据库结构变更的方式。每个迁移文件包含 up（应用变更）和 down（回滚变更）两个方法，按时间顺序命名（如 202503250001_create_users.js）。Knex 会自动记录已执行的迁移，确保每个迁移只运行一次。

    2） 技术选型
    使用 Knex.js 作为查询构建器和迁移工具，配合 pg 驱动连接 PostgreSQL。

    3）部署步骤
    安装依赖
    bash
    npm install knex pg
    配置 knexfile.js
    在项目根目录创建配置文件，指定数据库连接、迁移目录、种子目录等：

    创建迁移文件
    使用命令 npx knex migrate:make create_users 生成空白迁移文件，然后编写 up 和 down 方法。
    运行迁移

    bash
    npx knex migrate:latest --env development
    执行后，Knex 自动创建 knex_migrations 和 knex_migrations_lock 表，记录迁移历史。

    种子数据
    使用 npx knex seed:make initial_data 创建种子文件，在 seeds/initial_data.js 中插入初始数据（如默认徽章），然后运行：

    bash
    npx knex seed:run --env development
    4）团队协作与部署
    将所有迁移文件、种子文件及 knexfile.js 提交到 Git 仓库。

    团队成员拉取代码后，只需运行 npm install 安装依赖，再执行 npx knex migrate:latest 即可同步数据库结构。

    生产环境部署时，通过 CI/CD 流水线自动执行迁移命令，确保环境一致。
## 心得体会
    在本次 Moveup 项目的架构设计过程中，我收获良多，主要体现在以下几个方面：

    1. 分层架构的清晰化  
    最初我习惯将所有代码堆在一起，经过这次设计，我深刻理解了“分层”的意义：客户端层、网关层、业务服务层、存储与异步层各司其职，不仅让系统职责分明，还便于后续扩展和维护。通过 Mermaid 绘制架构图，我学会了用可视化工具表达设计思想，让团队沟通更高效。

    2. 数据库设计的版本控制  
    以前我都是手动执行 SQL 建表，经常出现团队成员数据库结构不一致、回滚困难的问题。这次我引入了 Knex 迁移脚本，将表结构变更以代码形式管理，配合种子数据，实现了开发、测试、生产环境的统一。我不仅掌握了迁移的基本用法，更认识到“基础设施即代码”的重要性。

    3. 技术选型的权衡
    在选择数据库时，我原本只考虑 MySQL，但结合运动软件的需求，最终确定 PostgreSQL + InfluxDB 的组合，利用 JSONB 和时序数据库分别处理非结构化数据和海量轨迹点。这种“用对的技术解决特定问题”的思维，让我在后续设计其他模块时也更加理性。

    4. 前后端分离的实践  
    虽然前端采用原生 Android，后端用 Node.js，但通过清晰的 API 设计、统一的错误响应格式，两者能够高效协作。我学会了如何设计 RESTful 接口，并考虑到移动端弱网环境下的数据同步策略。

    5. 文档与协作 
    通过编写架构文档、ER 图、CLAUDE.md 等，我意识到规范化的文档不仅是对自己思路的梳理，更是团队协作的基石。特别是在使用 dbdiagram.io 绘制 ER 图后，我能快速向他人展示数据库关系，减少了沟通成本。

    6. 工具链的整合  
    从 Mermaid 画图、dbdiagram.io 设计 ER 图，到 Knex 管理迁移，再到 Git 进行版本控制，这一整套工具链让我体会到了现代开发的自动化与标准化。未来我会更加主动地引入这类工具，提升开发效率。

    总之，这次架构设计让我从“只会写代码”向“能设计系统”迈出了一大步。我深刻认识到，好的架构不仅是技术的堆砌，更是对业务、团队、可维护性等多方面平衡的结果。这些收获将指导我今后的项目实践。
