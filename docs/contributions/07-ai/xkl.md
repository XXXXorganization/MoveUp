## 我完成的工作

### 1. AI 功能
- **功能类型**: 智能客服与数据分析（Agent 智能体自动执行计划排表）
- **使用模型**: Qwen/Qwen2.5-7B-Instruct (通过硅基流动 API 平台调用国内可用模型)

### 2. 实现内容
- [x] **后端 API**: 在 Node.js (`server.js` & `index.js`) 中独立封装了 `/v1/ai/chat` 接口，将大模型 API Key 安全存放在服务端。实现了深度上下文注入功能，后端会自动读取用户的历史跑步数据与周计划表拼接成 Prompt 发送给 AI。同时，编写了核心的后端拦截逻辑，利用正则表达式精准提取 AI 返回的 JSON 暗号（`###PLAN:[...]`），并自动将其写入数据库，完成了 AI 从“对话”到“动作执行”的闭环。
- [x] **前端调用**: 在 Android 客户端 (`AItalk.java` & `activity_aitalk.xml`) 实现了与后端的对接。构建了带有气泡 UI 的聊天界面，采用异步子线程进行 HTTP 轮询请求，支持用户通过自然语言查询自身数据或下达排表指令。
- [x] **错误处理**: 针对网络与大模型幻觉进行了多层容错处理。在前端加入了网络连接异常拦截与 UI 等待状态；在后端针对大模型 JSON 格式输出不规范（如末尾携带多余中文字符）的问题，通过正则匹配增强了容错提取能力，并设置了 15000ms 的超时保护。

## 心得体会
在本次 AI 功能的集成过程中，我经历了从“前端直接调大模型 API”到“后端封装代理层并结合私有数据”的架构升级，深刻体会到了企业级应用中密钥安全和系统解耦的重要性。

最大的技术收获在于对 AI Agent（智能体）工作流的探索。最初我只把大模型当做纯聊天的工具，但在实际开发中，我通过在系统提示词中设定严格的输出格式，配合后端的正则解析引擎，成功让 AI 具备了修改项目数据库的能力（自动添加跑步计划）。同时，在解决 AI 输出格式不可控（如 JSON 解析失败）的 Bug 时，我学会了不能完全依赖模型的“听话程度”，必须在后端代码中做好强大的异常兜底逻辑。这次实践让我对如何将 AI 能力真正落地到业务场景有了极大的信心。
